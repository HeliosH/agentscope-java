## 前言

如果你用过元宝的 SKILL 功能，让它帮你写代码、处理文件、跑脚本，你有没有想过：AI 执行完之后，那些文件去哪了？

这个问题背后，藏着一个工程问题：当 AI Agent 越来越"能干"，对底层存储要求越来越高时，如何通过升级存储后端与优化客户端配置，将单桶 LIST QPS 提升 100 倍、平均延迟降低 75%，支撑元宝 SKILL 沙箱的高并发访问。

## 一、沙箱里的 AI，需要一个持久化的空间

AI Agent 在执行任务时，通常运行在一个叫做"沙箱"（Sandbox）的隔离环境里——一个临时搭起来的工作台：东西可以放上去，代码可以在上面跑，但任务结束工作台收走，上面的东西也就没了。

这种设计的初衷是安全：沙箱里跑的代码出不去，外面也进不来，用户之间互相隔离。但 AI Agent 的任务越来越复杂：

- 处理用户上传的文件，完成分析后还要把结果交还给用户；
- 跨多轮对话持续工作，"记住"上一轮做到哪了、生成了哪些中间文件；
- 多个沙箱实例并行跑，需要共享同一份工作目录里的数据；
- 任务失败或中断后能够恢复，而不是从头再来。

这些需求有一个共同点：数据不能只活在沙箱里，它必须能被持久化保存下来，在沙箱销毁后依然存在，在下次需要时可以被访问到。换句话说，**沙箱需要一个路径来持久化保存数据**。

云原生时代，这个持久化空间的最优解就是对象存储。对象存储采用扁平化命名空间，可横向扩展至 PB 甚至 EB 级别，并提供高达 99.9999999999%（12 个 9）的数据设计持久性——即使沙箱所在宿主机宕机，Agent 产出物和关键数据依然高可用、不丢失。

但对象存储原生采用 RESTful API（如 S3 的 PUT/GET/DELETE），而沙箱内的 Agent 通常使用操作系统的标准文件系统调用（open/read/write）。直接使用对象存储，开发者必须修改业务代码显式调用 SDK/CLI 上传下载，增加开发成本、破坏代码通用性。

**腾讯云的选择**：把 COS 对象存储，通过 GooseFS MountPoint 挂载成沙箱内的本地目录，让沙箱像访问本地文件系统一样访问云端数据。

## 二、GooseFS MountPoint 是什么？它怎么跟沙箱配合？

GooseFS MountPoint（以下简称 MountPoint）是一款基于 FUSE（Filesystem in Userspace 用户态文件系统）框架开发的高性能客户端，专为腾讯云对象存储 COS 量身打造。核心能力：把 COS 对象存储"伪装"成一个本地目录，对上层应用完全透明。

挂载流程：

1. 宿主机收到创建沙箱请求后，生成唯一会话 ID（如 `sess_43ac581ce0b74d94`）；
2. 在 MountPoint 挂载目录 `/mnt/` 下创建该沙箱目录 `/mnt/sess_43ac581ce0b74d94/`——此 `mkdir` 直接穿透到 COS 存储桶，在桶里创建对应目录前缀；
3. 宿主机启动沙箱运行时，把虚拟文件系统挂载点下那个带哈希的长路径 bind mount 到一个对用户友好的短路径（如 `/data/sess_a/`）；
4. 用户在容器内对该目录的任何读写，沿 `bind mount → 软链接穿透 → MountPoint → COS 存储桶` 完整链路透传，对用户完全透明。

各层职责：

- **路径映射**：宿主机复杂路径 → 用户可见简洁路径；
- **virtio-fs**：虚拟机内部与宿主机间的文件系统共享协议，性能优于 NFS；
- **virtiofsd**：运行在宿主机，处理来自 MVM 的文件请求，同时承担安全隔离；
- **MountPoint**：把 COS 对象语义翻译成 POSIX 文件系统语义。

对 Agent 来说这一切透明，它只管读/写文件。

## 三、性能挑战：List 请求把存储打爆了

高并发 Agent 场景下，瞬时拉起成百上千个沙箱实例，发现沙箱启动变慢、对挂载路径下文件访问延迟巨大。经排查，是大量发给 COS 的 LIST 请求被频控。

**当前版本 COS，单个存储桶的 List 类请求默认频控阈值是 1000 QPS。** 听起来不少，对象访问场景完全够用，但在文件系统场景里远远不够。原因——对象存储和文件系统在设计哲学上有根本差异：

- 对象存储天然是扁平的 KV 结构，靠 key 取，LIST 是低频管理操作；
- 文件系统完全不同，每一次 `stat`、`ls`、`open`、路径遍历，背后都可能触发多次 LIST 或 HEAD 请求。

具体来说，访问路径 `/a/b/c/d/file` 时，操作系统内核需从根目录逐级确认 `/a`、`/a/b`、`/a/b/c`、`/a/b/c/d` 是否存在、每级是文件还是目录。MountPoint 为实现 POSIX 语义，对每一级路径节点都需要发查询——未知类型路径默认同时发 HEAD 和 LIST，等两者都返回才能得出结论。

**更糟糕的情况：访问不存在的路径。** 对象存储没有"目录"概念，目录是隐式的（靠对象 key 前缀模拟）。因此判断一个路径是否存在，必须兜底走一次 `ListObjects(prefix=xxx/, max-keys=1)` 才能确认，无法像本地文件系统那样靠 dentry 缓存拦截。这种"不存在路径的探测"在沙箱场景特别常见：训练任务找 checkpoint、业务代码做存在性检查……大量命中"不存在"，每次必须打穿到 COS 后端。

再叠加沙箱规模：元宝 SKILL 沙箱在 k8s 上部署，生产环境成百上千实例同时运行共用同一存储桶。单个沙箱请求量不大，汇总到桶级 QPS 轻松超过 1000。一旦频控，一连串雪崩：`COS 返回 503 SlowDown → MountPoint 客户端触发指数退避 → 文件操作卡顿 → ls 变慢、stat 超时 → 任务卡住 → 最终报 I/O error`，用户直接感受到 AI 工具"卡死了"或"报错了"。

**根因不是某个具体 bug，而是两种存储范式之间的天然错配**——对象存储被用于文件系统场景时，原本低频的 LIST 变成高频操作，而现有 COS 架构在设计上没预见这种用法。

## 四、解法：换新引擎，串行改并行，减少无效请求

两个方向：提升后端能力 + 减少不必要的前端请求。

### 4.1 接入 newCOS：从根本上提升 List 性能

newCOS 是腾讯云对象存储的全新升级架构。**YottaIndex** 是 newCOS 的索引存储子系统——不存放对象二进制内容，只负责对象的逻辑标识、元数据和路由信息，专为 newCOS 量身定制，专注于把对象存储索引做到极致。针对 List 场景，YottaIndex 从五个维度端到端优化：

1. **索引列瘦身**：拆分索引字段，列表查询仅读取少量轻量数据，大幅减少数据读取量与磁盘开销；
2. **原生支持目录能力**：底层适配目录逻辑，查询子目录无需遍历全部文件，大幅减少数据扫描量；
3. **数据有序存储**：不打散数据分片，列表查询可顺序读取，无需跨分片拼接结果；
4. **热点快速弹性扩容**：小分片设计，检测到流量热点后秒级完成扩容，应对突发高并发；
5. **智能优化各类复杂场景**：自动跳过空分片、预估分片大小并自适应并发，同时支持超时断点续查，规避查询卡顿。

压测验证（全部使用**完全不存在的路径**构造请求——刻意模拟最恶劣真实场景，大量"负缓存穿透"，不命中本地缓存）：

| 指标 | 老 COS | 接入 newCOS 后 |
|------|--------|----------------|
| LIST QPS | 1,000 | 100,000（提升 100 倍） |
| LIST 平均延迟 | 基准 | 降低约 75% |

### 4.2 MountPoint 配置调优：让每次查询少一点浪费

**问题一：串行查询导致的相互阻塞。** MountPoint 对未知类型路径的默认 lookup 流程是先发 HEAD、再发 LIST，串行执行。HEAD 遇流控触发指数退避时，LIST 也跟着被阻塞，整体 QPS 上不去（"最长者阻塞"）。解法——开启并行 lookup：

```
disable-parallel-lookup-unknown-type: false   # false = 开启并行
```

开启后 HEAD 和 LIST 同时发出，不再互相等待。即使 HEAD 遇流控，LIST 也能继续推进。

**问题二：沙箱场景下不存在"显式目录"。** 普通 COS 场景中目录有时是用户显式创建的（创建以 `/` 结尾的空对象），MountPoint 默认会先检查"显式目录"是否存在，多发一次 HEAD。但沙箱场景目录全部隐式，不存在任何显式创建的目录对象，这次 HEAD 多余。解法——关掉该选项：

```
lookup-explicit-dirs-first: false   # 沙箱桶内无显式目录，省掉多余 HEAD
```

两参数叠加效果：

- 每次路径查询的 COS 请求从"3 次串行"变成"2 次并行"：
  - 原来：`HEAD → HEAD(查显式目录) → LIST`，三步串行，任一步流控阻塞后续；
  - 现在：`HEAD + LIST` 同时发出，完全并行，只需等较快那个返回；
- HEAD 请求数量减半，流控压力直接降低 50%；LIST 也不再被 HEAD 延迟拖累，整体吞吐显著提升。

## 五、现在这套方案在哪里跑着？

基于 MountPoint + newCOS 的沙箱存储方案已在以下业务生产环境稳定运行：

- **元宝 SKILL 沙箱**：本次压测和优化的直接受益方。面向用户的 AI Agent 功能，支持代码执行、文件处理、工具调用等复杂任务，对沙箱存储并发能力和稳定性要求最高；
- **ima 沙箱**：AI 助手沙箱环境，需在对话过程中持久化用户文件、跨轮次共享工作状态；
- **某外部客户**：企业 Agent 沙箱接入场景，对数据隔离和存储可靠性有严格要求。

三个业务覆盖从内部 AI 产品到外部客户服务的不同场景，验证了方案在不同负载特征和安全要求下的通用性。

## 后记

回顾整个过程，几点总结：

- **对象存储 ≠ 文件系统，但它们可以融合。** 两种存储范式有根本差异，强行把对象存储当文件系统用会暴露很多问题；但通过客户端（MountPoint）和后端存储服务（newCOS）的精心配合、针对性参数调优，可以让两者协同工作，发挥各自优势。
- 随着 AI Agent 场景快速普及，沙箱存储挑战会越来越多。高并发下的频控问题只是开始，后续还有数据一致性、冷热分层、跨地域访问等更复杂问题。

## 附：对本方案的意义（编者注）

本仓库的存储设计**不走 FUSE 路线**，采用框架原生"无 FUSE 两层读"架构（`SandboxBackedFilesystem` + `RemoteFilesystem` + `BaseStore`）。因此本文描述的 FUSE 引发的 LIST 放大雪崩，在本方案中**根本不发生**：

- 不挂对象存储为文件系统 → 不会有 POSIX→LIST 翻译放大；
- `RemoteFilesystem` 走 `BaseStore.get(namespace, key)` 是**显式 KV 查询**，一次 `get` 一次 KV 操作，无逐级目录探测；
- "不存在路径"在 KV 模型里就是一次 `get` 返空，**天然无穿透放大**。

本文真正值得借鉴的是其**容量与索引设计哲学**（newCOS YottaIndex 的"索引列瘦身 / 原生目录 / 有序存储 / 小分片弹性 / 复杂场景优化"五维思路），映射到本方案即 [13-storage-upgrade.md](./13-storage-upgrade.md) S5：**Postgres 存元数据索引、MinIO 存大对象，目录列举走索引而非高频 LIST**。详见该文档"明确不做"与 S5 节。
